# Инструкция по запуску Story Factory (Метод Тимочко)

Эта система помогает быстро собирать Instagram Stories по схеме **Hook → Context → Value → CTA**.

- **Gemini 1.5 Pro** генерирует сценарий и структуру сторис.
- **Google Vids** используется для видео-слайдов.
- **Streamlit** дает простой веб-интерфейс, чтобы управлять процессом без отдельного фронтенда.
- Для статичных слайдов предусмотрен fallback-пайплайн и режим работы **со своими фото**.

---

## Что такое Streamlit простыми словами

Streamlit — это способ сделать веб-интерфейс прямо на Python.

То есть вместо отдельного сайта на React / Vue ты запускаешь одну команду и получаешь локальную страницу, где можно:

- ввести тему сторис,
- выбрать аудиторию и стиль,
- загрузить свои фото,
- получить сценарий,
- отредактировать каждый слайд,
- запустить обработку.

Для этого проекта Streamlit — самый быстрый путь к рабочему MVP.

### Streamlit — это бесплатно или потом просит деньги?

Сам **Streamlit как open-source библиотека — бесплатный**.

То есть:

- если запускать локально на своем компьютере — это бесплатно,
- если поднимать на своем VPS — ты платишь только за сервер,
- деньги появляются не из-за Streamlit как такового, а из-за **хостинга**, **API**, **хранения файлов** и возможных платных сервисов вокруг.

Иными словами: **Streamlit сам по себе не “сажает на платную подписку”**. Платить ты начинаешь только тогда, когда выносишь сервис в облако или подключаешь платные AI/API-компоненты.

---

## Почему я выбрал Streamlit, а не другой веб-стек

Честно: в этой сессии я **не делал сравнение по 100 внешним источникам**. Выбор сделан как инженерное решение под твой конкретный кейс.

Почему именно он сейчас уместен:

1. **Очень быстро собрать MVP** — форма, загрузка фото, редактор, кнопки, предпросмотр.
2. **Всё на Python** — без отдельного фронтенда на React/Next.
3. **Удобно для внутреннего AI-инструмента**, где важнее скорость разработки, чем “идеальный SaaS UI”.
4. **Легко дальше переделать**: если сервис вырастет, backend-логику можно оставить, а UI потом вынести в отдельный фронтенд.

Если бы задача была сразу такая:

- много пользователей,
- биллинг,
- личные кабинеты,
- публичный SaaS,
- сложный фронтенд,

то я бы **не оставлял финальную архитектуру на Streamlit** и рекомендовал бы связку вроде **FastAPI + React/Next.js**.

Но для твоей текущей стадии Streamlit — это **не “идеальный навсегда”, а “самый рациональный быстрый старт”**.

## Где лучше хостить такой сервис

Тут важно разделить 2 сценария.

### Вариант 1 — сервис для себя / для своей команды
Лучший вариант сейчас:

- **локально на своем компьютере**, либо
- **на обычном VPS**.

Почему:

- у тебя есть загрузка файлов,
- есть генерация картинок,
- есть Playwright / Google Vids / сохранение сессии,
- есть зависимость от браузерной автоматизации.

Для такого стека **обычный VPS часто лучше, чем модный serverless/PaaS**.

### Вариант 2 — если потом делать платный сервис
Тогда я бы рекомендовал уже не “чистый Streamlit как конечную архитектуру”, а:

- **FastAPI backend**,
- **очередь задач**,
- **хранилище файлов**, 
- **отдельный frontend**,
- **авторизацию и биллинг**.

Потому что платный продукт требует:

- стабильности,
- очередей,
- фоновых задач,
- разделения пользователей,
- логов,
- контроля нагрузки.

### Что рекомендую практически

Если запускать в ближайшее время:

1. **Сейчас**: локально или VPS.
2. **Следующий этап**: VPS + Docker.
3. **Платный SaaS позже**: backend/frontend разделить.

### Что я НЕ рекомендую как основной вариант под этот кейс

- Streamlit Community Cloud как основную прод-среду для этого проекта,
- чистый serverless для пайплайна с Playwright и долгими задачами,
- усложнять стек раньше времени.

Потому что у тебя проект не “просто форма и табличка”, а сервис с:

- медиа,
- AI,
- локальными файлами,
- потенциально браузерной автоматизацией,
- длительными шагами обработки.

## Как теперь получить не просто план, а сами сторис

Теперь логика такая:

### Шаг 1 — Сгенерировать план
Кнопка **«Сгенерировать план сторис»** делает только сценарий.

### Шаг 2 — При необходимости отредактировать слайды
Можно поправить:

- текст,
- тип слайда,
- промпт,
- инструкции,
- привязку фото.

### Шаг 3 — Нажать **«🍌 Создать AI сторис (Nano Banana Pro)»**
Эта кнопка запускает **реальную AI-генерацию** через модель `nano-banana-pro-preview`. 

**Что нового:**
- Добавлен **индикатор прогресса**. Теперь ты видишь, какой именно слайд сейчас генерируется (например, "Генерация слайда 2 из 5"), чтобы не казалось, что программа зависла.
- В отличие от простого наложения текста, здесь AI сам рисует композицию по промпту.
- Если загружено фото, AI использует его как основу (Image-to-Image).
- Текст вписывается в саму картинку средствами нейросети.

Результаты сохраняются в:
```bash
outputs/nano_banana_story_exports/...
```

### Шаг 4 — Локальный fallback PNG
Если нужно просто быстро наложить текст на фото без AI-генерации самой картинки, используй кнопку **«🧩 Локальный fallback PNG»**.

### Шаг 5 — Если нужны именно видео-сцены
Тогда используется кнопка внешнего пайплайна / Google Vids.

То есть теперь в приложении есть 2 разных действия:

1. **План** — сценарий.
2. **PNG сторис** — уже готовые статичные сторис.
3. **Google Vids / внешний пайплайн** — отдельная ветка для видео.

## Как работает система

### 1. Gemini генерирует драматургию сторис
Ты задаешь цель, например:

> «Продать курс по ИИ-автоматизации для новичков»

Gemini строит серию слайдов по методу Тимочко:

1. **Hook** — остановить скролл
2. **Context** — объяснить контекст
3. **Value** — показать пользу / смысл / доказательства
4. **CTA** — подвести к действию

### 2. Ты редактируешь план в веб-интерфейсе
В `app.py` можно:

- поправить тексты,
- поменять тип слайда (`photo` / `video`),
- изменить визуальные промпты,
- назначить конкретные фото на конкретные слайды.

### 3. Запускается обработка слайдов
- Для `video`-слайдов используется Google Vids automation.
- Для статичных слайдов сейчас работает placeholder/fallback-логика, в которую можно подставлять свои изображения.

---

## Режимы работы с визуалами

### 1. AI-визуалы
Подходит, если своих фото нет.

Система делает упор на AI-сцены, визуальные промпты и видео-генерацию.

### 2. Свои фото
Подходит, если у тебя уже есть нужные изображения.

Ты загружаешь фото в интерфейс, а затем распределяешь их по слайдам. Это особенно полезно для личного бренда, кейсов, отзывов, продукта, бэкстейджа.

### 3. Смешанный режим
Самый практичный вариант.

Например:
- Hook и Context — на твоих живых фото,
- Value и CTA — на AI-визуалах или видео.

## Подготовка окружения

### 1. Установи зависимости
```bash
python3 -m pip install google-generativeai playwright python-dotenv streamlit
python3 -m playwright install chromium
```

> Если `streamlit` не находится как команда, используй запуск через `python3 -m streamlit`.

### 2. Настрой `.env`
В корне проекта создай файл `.env`:

```env
GOOGLE_API_KEY=ВАШ_GEMINI_API_KEY
GEMINI_MODEL=models/gemini-2.5-flash
```

Теперь `story_orchestrator.py` сам загружает `.env` при старте.

`GEMINI_MODEL` не обязателен. Если его не указывать, приложение само попробует выбрать совместимую модель из доступных в твоем аккаунте.

### 3. Сохрани авторизацию Google для Vids
```bash
python3 google_content_factory/save_auth.py
```

Откроется браузер. Войди в нужный Google-аккаунт и сохрани сессию.

После этого должен появиться файл `google_auth.json`.

## Как запустить веб-интерфейс

```bash
python3 -m streamlit run app.py
```

После запуска откроется локальная страница, где можно:

1. ввести цель сторис,
2. указать аудиторию,
3. выбрать стиль,
4. выбрать режим визуалов,
5. загрузить свои фото,
6. сгенерировать план,
7. вручную отредактировать каждый слайд,
8. запустить обработку.


## Важные замечания

### 1. Google Vids может меняться
Автоматизация через Playwright зависит от интерфейса Google. Если Google поменяет кнопки, названия элементов или поведение страницы, селекторы придется подправлять.

### 2. Статичные слайды пока на fallback-логике
Сейчас для статичных слайдов реализован placeholder-процесс. Он уже понимает, что на слайд можно назначить пользовательское фото, но полноценную генерацию через отдельный image API можно добавить следующим этапом.

### 3. Если ключ не найден
Проверь, что:

- файл `.env` лежит в корне проекта,
- в нем есть `GOOGLE_API_KEY=...`,
- ключ рабочий.

### 4. Если Gemini пишет, что модель не найдена
Некоторые старые названия моделей Gemini больше не поддерживаются в текущем API.

В текущей версии `story_orchestrator.py` уже добавлен автоматический fallback: система сначала смотрит список реально доступных моделей в твоем аккаунте, а потом пробует совместимые варианты по очереди. Поэтому если старое имя вроде `gemini-1.5-pro` больше не работает, приложение постарается автоматически перейти на актуальную модель.

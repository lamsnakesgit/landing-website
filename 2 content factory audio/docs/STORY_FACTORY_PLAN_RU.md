# План развития Story Factory — по-человечески и по делу

## Что уже сделано

На текущий момент уже есть рабочая база:

- исправлена загрузка `GOOGLE_API_KEY` из `.env`,
- добавлен `app.py` на Streamlit,
- есть режимы **AI-визуалы / свои фото / смешанный**,
- можно генерировать план сторис,
- можно редактировать слайды вручную,

То есть это уже не просто идея, а **MVP-прототип**, который можно развивать до нормального рабочего инструмента.

---

## Streamlit — это что вообще?

Если совсем просто:

**Streamlit — это Python-библиотека / микро-фреймворк для быстрого создания веб-интерфейсов.**

То есть не нужно отдельно писать фронтенд на React, Vue и так далее.
Можно прямо на Python сделать страницу с:

- полями ввода,
- кнопками,
- загрузкой файлов,
- таблицами,
- JSON-предпросмотром,
- редактором данных.

Для этого проекта Streamlit подходит идеально, потому что нам нужен **не красивый маркетинговый сайт**, а **быстрый рабочий интерфейс управления генерацией сторис**.

Итого:

- как **библиотека** — да,
- как **фреймворк для быстрых внутренних веб-интерфейсов** — тоже да.

Если по-простому: **это самый быстрый способ собрать рабочую веб-морду на Python**.

## Сколько это займет по человеко-часам

Ниже — честная оценка, без розовых очков.

### 1. Чтобы работало «уже нормально локально»
**Оценка: 6–12 человеко-часов** от текущего состояния.

Что сюда входит:

- стабилизация Streamlit-интерфейса,
- более аккуратная обработка ошибок,
- улучшение режима своих фото,
- сохранение/загрузка проектов,
- нормальные статусы шагов,
- базовая проверка сценариев перед запуском.

Это уровень:

> «Я могу сам открыть интерфейс, задать тему и относительно быстро получить сторис-пайплайн без лишней боли».

### 2. Чтобы можно было «нормально поручать сторис и быстро получать результат»
**Оценка: 12–24 человеко-часа**.

Что нужно добавить:

- более надежную логику работы с Google Vids,
- очередь задач / последовательную обработку,
- более понятное разруливание слайдов с фото,
- сохранение версий сценария,
- шаблоны под разные типы прогрева / продаж / кейсов,
- предпросмотр структуры до запуска,
- более стабильную работу с загрузками файлов.

Это уровень:

> «Можно реально использовать как рабочий инструмент, а не как демку».

### 3. Чтобы оставить систему «почти автономно»
**Оценка: 24–45 человеко-часов**.

Что сюда входит:

- логирование,
- автосохранение проектов,
- фоновая обработка,
- понятные ретраи и fallback-сценарии,
- хранение результатов,
- контроль статуса задач,
- защита от типовых падений,
- доработка AI/фото-режима,
- более надежная интеграция с внешними сервисами.

Это уже уровень:

> «Систему можно надолго оставить делать рутинную часть, а человек только проверяет результат и творчески правит».

### 4. Если нужен почти продакшн / удаленная работа / “с телефона”
**Оценка: 40–80+ человеко-часов**.

Сюда уже входят:

- серверная архитектура,
- хранение задач и медиа,
- авторизация,
- удаленный доступ,
- уведомления,
- API / бот / очередь,
- нормальный деплой,
- мониторинг и восстановление после ошибок.

## Насколько меня можно оставить автономно?

Короткий ответ:

**На разработку — сильно. На авторизации, доступах и творческом финальном контроле — не полностью.**

### Что я могу делать автономно

Без постоянного дергания тебя я могу:

- проектировать структуру решения,
- писать Python-код,
- собирать Streamlit-интерфейс,
- делать рефакторинг,
- писать документацию,
- готовить пайплайны,
- подключать и чинить локальную логику,
- делать git commit,
- проверять запуск,
- править сценарии по правилам,
- улучшать UX внутри текущего стека.

### Где нужен человек

Нужен ты, когда дело касается:

- логина в Google,
- CAPTCHA / 2FA,
- платных ключей / доступов,
- выбора из нескольких креативных направлений,
- утверждения итоговой сторителлинг-стратегии,
- реальных бизнес-решений типа «что именно продаем первым». 

### Реалистичная модель автономности

Самый удобный формат такой:

1. Ты даешь цель.
2. Я автономно делаю блок работ на **1–3 часа**.
3. Показываю промежуточный результат.
4. Если нужен логин, подтверждение или выбор направления — подключаешься.
5. Дальше снова можно оставить меня допиливать следующий блок.

То есть **автономность высокая**, но это не магия “оставил на ночь и всё в мире само заработало без доступов и без человеческого решения”.

**Да.**

Могу:

- писать документацию по-русски,
- комментировать изменения по-русски,
- вести рабочий процесс по-русски,
- делать понятные инструкции без техно-каши.

Если хочешь — вообще весь дальнейший контур можно держать полностью на русском.

---

## Буду ли я делать коммиты?

**Да.**

Я уже могу:

- вносить изменения,
- делать осмысленные git commit,
- держать изменения по этапам,
- не трогать лишние файлы,
- подписывать коммиты понятными сообщениями.

Нормальный рабочий режим:

- один смысловой блок работы,
- один аккуратный commit.

## Что значит ошибка Cline / 503?

Ошибка такого вида:

```json
{"message":"503 status code (no body)","status":503,"modelId":"gemini-3-flash-preview","providerId":"openai"}
```

Обычно означает **временную проблему на стороне провайдера / шлюза / модели**, а не ошибку именно в твоем проекте.

### Простыми словами

Код `503` — это обычно:

- сервис временно недоступен,
- перегружен сервер,
- отвалился промежуточный gateway,
- модель временно не отвечает,
- у провайдера сбой маршрутизации запроса.

### Важно

Это **не выглядит как баг Streamlit**.
И это **не похоже на ошибку именно твоего `app.py`**.

Это больше похоже на:

- временный сбой у модели,
- временный сбой у провайдера,
- проблему в маршруте `Cline -> provider -> model`.

### Что делать, если видишь такое

1. Просто подождать и повторить.
2. Переключить модель на более стабильную.
3. Проверить, не используется ли preview-модель.
4. Проверить настройки провайдера в Cline.
5. Если ошибка повторяется массово — это уже не локальная проблема проекта.

### Если совсем по делу

`503` в Cline сейчас = **скорее всего внешний временный сбой**, а не «мы сломали код Story Factory».

## Рекомендуемый план следующих работ

### Этап 1 — довести MVP до удобного состояния
**6–12 часов**

Сделать:

- сохранение проектов,
- историю последних генераций,
- более чистый редактор слайдов,
- нормальный статус по каждому слайду,
- валидацию входных данных,
- улучшить логи ошибок.

### Этап 2 — усилить режим фото / AI / mixed
**6–10 часов**

Сделать:

- точное назначение фото по слайдам,
- автологику: какие слайды лучше делать на фото, какие на AI,
- шаблоны композиций под фото,
- отдельные инструкции для статичных сторис.

### Этап 3 — сделать систему реально удобной в ежедневной работе
**8–15 часов**

Сделать:

- сохранение сценариев,
- пресеты под тип контента,
- удобный перезапуск отдельных слайдов,
- экспорт результатов,
- аккуратные логи,
- предсказуемое восстановление после ошибки.

# Docker для начинающих: Простая аналогия

Часто возникает путаница между двумя главными концепциями Docker: `docker-compose.yml` и `Dockerfile`. Зачем их два и почему нельзя всё запихать в один файл?

Давайте представим систему как **Строительную площадку**.

## 1. Файл `docker-compose.yml` — это Главный Прораб (Оркестратор)
Этот файл отвечает только за **управление** (оркестрацию).
* Ему всё равно, из чего сделан кирпич. Ему важно только то, **куда его поставить** и **с кем соединить**.
* Он понимает строгий язык YAML (структурированные списки). В нём нельзя писать команды системы (типа `apt get install` или `cd /folder`).
* **Его задачи:**
  - Скачать готовую программу из интернета (например, `image: n8nio/n8n`).
  - Прокинуть нужные порты наружу (чтобы сайт открывался на 443 порту).
  - Сказать, чтобы база данных (Postgres) и приложение (WhatsApp API) общались друг с другом по одной закрытой сети.

## 2. Файл `Dockerfile` — это Шеф-повар (Инструкция по сборке)
Этот файл отвечает за **внутренности одной конкретной программы**.
* Это рецепт, по которому докер "готовит" (собирает) уникальный контейнер перед тем, как отдать его Прорабу.
* Он понимает прямые команды операционной системы (`RUN`, `COPY`, `FROM`).
* **Его задачи:**
  - Взять за основу готовую оболочку (например, чистый Linux или чистый n8n): `FROM n8nio/n8n:latest`
  - Дать права администратора: `USER root`
  - Скачать и установить нужные плагины внутрь оболочки: `RUN apk add --no-cache ffmpeg`
  - Вернуть права обратно и завершить сборку.

---

## Как это работает вместе? (Случай с n8n + FFmpeg)

По умолчанию создатели n8n выкладывают в интернет **чистую сборку**, в которой нет встроенных утилит по типу FFmpeg. Если вы просто пропишете в `docker-compose.yml` строку `image: n8nio/n8n`, ваш Прораб тупо скачает эту "голую" версию с интернета и запустит её. FFmpeg в ней работать не будет.

Чтобы исправить это, мы делаем следующее:
1. Мы создаем свой рецепт (`Dockerfile`) и пишем в нем: *"Возьми чистый n8n и установи в него FFmpeg"*.
2. Мы идем к Прорабу (`docker-compose.yml`) и заменяем инструкцию "скачай из интернета" (`image`) на инструкцию "приготовь на месте" (`build: .`).
3. Теперь при запуске сервера `docker compose up -d --build`, система сначала приготовит уникальный "прокачанный" образ по вашему рецепту, а потом отдаст его Прорабу, чтобы тот разместил его в рабочей сети.

### Итог
- **`docker-compose.yml`** — Настройки сети, портов и того, какие программы вообще должны работать на сервере.
- **`Dockerfile`** — Чертеж модификации (прокачки) одной конкретной программы перед её запуском.

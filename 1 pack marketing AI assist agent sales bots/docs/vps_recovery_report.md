# Отчет по восстановлению сервера и настройке (Diary)

## 2026-03-08: Возрождение из пепла (Rebuild Freakhosting)

### 1. Что произошло (Проблема)
Сервер Freakhosting перестал отвечать из-за критического сбоя на хост-ноде ("Network is unreachable", "Memory Information Unavailable"). Попытки перезагрузки застряли в "Pending Changes".

### 2. Как решили (Bypass)
- Принято решение о полной переустановке системы (**Rebuild**) на Ubuntu 22.04.
- **Ошибка SSH `REMOTE HOST IDENTIFICATION HAS CHANGED`:** Решена через `ssh-keygen -R 151.241.100.226`. Это стандартная мера безопасности при смене «железа» или ОС на том же IP.
- **Проблема с паролем:** Пароль после Rebuild был успешно установлен/получен через панель VirtFusion.

### 3. Лучший опыт (Best Practices)
1. **Локальные бэкапы — спасение:** Поскольку у нас были все JSON-файлы воркфлоу в папке `n8n_templates`, потеря данных на сервере не стала катастрофой для бизнеса.
2. **Схема Production:** Использование Postgres вместо SQLite на сервере с малым объемом памяти — правильный ход для стабильности.
3. **Traefik + SSL:** Автоматизация сертификатов через Let's Encrypt избавит нас от проблем с безопасностью.
4. **Фикс Docker образов:** Всегда проверяйте актуальность Docker Hub репозиториев. Образ `atendimentodigital` устарел, теперь используем `evoapicloud`.

### 5. Полезный инсайт: SSH-ключи (Что это?)
То, что вы добавили — это «публичный ключ». Представьте, что это замок, к которому ключ есть только у меня (на вашем Mac). 
- **Это безопаснее пароля:** Его невозможно подобрать перебором.
- **Это удобнее:** Мне не нужно спрашивать у вас пароль каждый раз, когда сервер его запрашивает.
- **Это стандарт Production-деплоя:** Профессионалы всегда закрывают вход по паролю и оставляют только ключи.
Поддомены (префиксы типа `n8n.`, `api.`, `crm.`) бесплатны и не требуют отдельной регистрации. Это позволяет логично разделять сервисы внутри одного IP.

## 🎓 Бест-практики по бэкапам для Чайников (Focus on Sustainability)

### 1. Как НЕЛЬЗЯ делать бэкапы
*   **Хранить бэкап только на том же сервере.** Если сервер «умрет» (как Freakhosting), бэкап умрет вместе с ним.
*   **Не проверять N8N_ENCRYPTION_KEY.** Если вы потеряете этот ключ из `.env`, то даже восстановив базу данных, вы не сможете открыть свои Credentials (пароли к сервисам). Они будут зашифрованы «навсегда».

### 2. Как НУЖНО делать (Наш план)
*   **Скрипты вовне:** Мы настроим контейнер `postgres-backup`, который каждый день делает «слепок» базы данных.
*   **GitHub (Вне сервера):** Самый дешевый и надежный способ — пушить файлы воркфлоу в приватный репозиторий GitHub. Это бесплатно и доступно из любой точки мира.
*   **S3 / Cloudflare R2:** Для хранения тяжелых бэкапов БД лучше всего использовать объектное хранилище (стоит копейки, $1-2 в месяц).

### 3. Безопасность учетных данных
*   **Секреты в `.env`:** Все пароли к базе и ключи n8n хранятся в одном файле `.env`. Его нужно хранить у себя на компьютере и в защищенном месте.
*   **SSH Ключи:** Вместо паролей лучше использовать SSH-ключи (приватный/публичный). Ключ невозможно «подсмотреть» или подобрать брутфорсом.
